今回はその内容の要点を抽出した上でわかりやすく噛み砕いて解説します。
元ポストはこちら：
https://x.com/zodchiii/status/2048345453096313005
Skillsの構造はたった3つのパーツ
6つの法則に入る前に、Skillsの基本構造だけ押さえておきましょう。
ポケモンのわざマシンを思い浮かべてください。技の中身を知らなくても、装備するだけで使える。SkillsもSKILL.mdというファイルを1つ置くだけで、Claudeに新しい能力が追加されます。
Skillsとは、frontmatterと呼ばれる設定欄と、本文の指示内容で構成されたマークダウンファイルです。
置く場所は決まっています。
markdown
~/.claude/skills/スキル名/SKILL.md
"フロントマター"とは、ファイルの冒頭に書く「設定欄」のことです。名前と説明文をここに書くと、Claudeが自動で認識します。
実際のディレクトリはこんな感じです。


markdown
~/.claude/skills/
├── weekly-report/
│   └── SKILL.md
├── campaign-analysis/
│   └── SKILL.md
└── competitor-check/
├── SKILL.md
└── TEMPLATES.md
動き方もシンプルで、Claudeがフロントマターの説明文を読んで自動で呼び出すか、/スキル名で手動呼び出しするかの2パターンしかありません。
構造はたったこれだけです。問題は、この中身をどう書くか。
イメージとしては、新しく入ったメンバーにオンボーディング資料を作る感覚です。
「どんな仕事を任せるか」「仕事の進め方」「報告のフォーマット」「まず読むべき資料」「担当範囲」を整理して渡す。Skillsはそれと同じことをClaudeに対してやります。
100個のSkillsを分析した結果、「動くSkills」と「動かないSkills」を分ける6つの設計法則が浮かび上がりました。
説明文が短いとClaudeに無視される
■ 法則1: descriptionに「いつ使うか」を書く
マーケの現場で考えてみてください。
「SNS施策やって」と代理店に投げたら、何が返ってくるか分かりません。担当者が困るだけです。
「Instagramリール運用、25-34歳女性、認知目的、予算50万」。ここまで書いて初めて、期待通りのアウトプットが返ってきます。
Skillsのdescriptionも全く同じです。
Claudeは、登録されている全Skillsの説明文を常にスキャンして、どれを読み込むか判断しています。「何をするか」だけでは「SNS施策やって」と変わりません。Claudeが「いつこのSkillsを使えばいいのか」分からず、結果、無視されます。
実際のSkillsで見てみましょう。
動かない書き方:
markdown
description: "レポート作成"
たった数文字。これでは「いつ使うか」が分かりません。
動く書き方:
markdown
description: "キャンペーン終了後の週次レポートを生成する。
'レポート', '週次', '振り返り', 'KPI'
というキーワードで呼び出す。月曜朝の定例MTG前に使う"
ポイントは3つの要素を入れることです。
何をするか、いつ使うか、どのキーワードで反応するか。キャンペーンブリーフで「媒体」「ターゲット」「目的」を明記するのと同じ要領です。
データも明確に出ています。50文字未満のdescriptionは、呼び出し頻度が3-5倍低い。 ここで一気に差がつきます。さらに、Claudeが認識する説明文は最初の250文字が勝負です。ユースケースは先頭に書きましょう。
Claudeに任せたい仕事を定義できたら、次は仕事の進め方の伝え方です。
Skillsは「お願い」ではなく「発注書」
■ 法則2: 命令形で書く
代理店への発注書を想像してください。
「いい感じのバナーお願いします」。これだと何が返ってくるか毎回ガチャです。担当者の解釈次第で、出来上がりが全く変わります。
「1200x628px、CTAボタン赤、コピーは○○、背景白、3パターン納品」。ここまで書けば、期待通りのものが出ます。
Skillsも全く同じです。「お願い」ではなく「発注書」として書きます。
実際のSkillsで比べてみましょう。
動かない書き方:
markdown
今週のキャンペーンレポート、お願いできますか？
いい感じにまとめてもらえると嬉しいです。
あと改善点もあれば教えてください。
動く書き方:
markdown
先週のキャンペーンデータを集計せよ。以下をチェック:
1. CPA・ROAS・CVRの推移
2. 予算消化率と残予算
3. クリエイティブ別のCTR比較
表形式で出力。各指標に前週比を明記
Claudeは丁寧な依頼より、直接的な指示の方が確実に実行します。
Anthropic公式リポジトリに入っているSkillsも全て同じパターンです。直接的な動詞、番号付きステップ、明示的な出力形式。公式がそうしているなら、間違いありません。
業務手順書ができたら、次に決めるのは「報告のフォーマット」です。
出力形式を決めるだけで結果が安定する
■ 法則3: 出力フォーマットを指定する
週次のマーケレポートで考えてみてください。
「今週どうだった？」と聞くだけだと、報告者によって粒度がバラバラになります。ある人はKPIだけ、ある人は施策の話、ある人は感想文。毎回フォーマットが違うので、比較もできません。
「KPI表、前週比、要因分析3行、来週のアクション」。テンプレートを決めるだけで、誰が書いても同じ品質のレポートが出ます。
Skillsも全く同じ原理です。形式を決めるだけで、毎回同じ構造の出力が返ってきます。
実際のSkillsで見てみましょう。
動かない書き方:
markdown
週次レポートを作って
これだと、KPIだけの1行で返ってくることもあれば、10ページの詳細分析で返ってくることもあります。毎回バラバラです。
動く書き方:
markdown
週次マーケレポートを以下の形式で生成:
■ KPIサマリー
- CPA / ROAS / CVR を表形式で
■ 前週比
- 各指標の増減と要因を1行ずつ
■ トップ施策
- 最もCVRが高かった施策とその要因
■ 来週アクション
- 改善施策を3点、優先度付きで

形式を決めるだけで、毎回同じ構造の出力が返ってきます。Skillsが「再利用可能」になるのはこの瞬間です。
ここまでの3つの法則、どれも結局は「新しいメンバーに仕事を引き継ぐとき、何をどう伝えるか」を整理するのと同じなんですよね。
形式が決まったら、次は「まず既存のやり方を見てから動いて」のステップです。
「まず読め」の一行がSkillの品質を決める
■ 法則4: 実行前に既存ファイルを読ませる
新しいマーケ担当がキャンペーンを引き継いだ場面を想像してください。
いきなり企画書を作り始めたら、ブランドのトーンも、過去のNGパターンも知らない状態でアウトプットが出ます。当然やり直しです。
まず過去のキャンペーン資料、ブランドガイドライン、NGパターンを読む。それを踏まえて企画を作れば、ブランドに合ったものが一発で出ます。
Skillsも全く同じです。「まず既存ファイルを読め」の一行で同じ効果が出ます。
実際にはこう書きます。
動かない書き方:
markdown
来月のキャンペーン企画を作って
汎用テンプレートで企画が生成されます。自社のブランドやターゲットとは合いません。
動く書き方:
markdown
1. 過去3ヶ月のキャンペーン結果ファイルを読む
2. ブランドガイドラインを確認する
3. NGパターン集を参照する
4. ターゲット層の最新データを確認する
5. 上記を踏まえてキャンペーン企画を生成する
企画の範囲も具体的に指定します。ターゲット設定、チャネル選定、予算配分、KPI設計。企画生成後はブランドガイドラインとの整合性をチェックして、問題があれば修正する。ここまでワンセットです。
この「まず読め」の一行が、「汎用的だけど使えない出力」と「自分のプロジェクトにフィットした出力」を分けます。
ここまでで仕事の受け渡しに必要な基本が揃いました。次は直感に反するけど強力なパターンです。
高品質Skillsの70%が「やらないこと」を書いている
■ 法則5: 範囲外を明記する
「リード獲得はマーケ、商談化はセールス」。マーケとセールスの境界が曖昧だと、ボールが落ちます。どちらも「あっちがやると思ってた」で機会損失が発生する。よくある話です。
この境界を明確に定義するだけで、チームの動きは劇的に変わります。
Skillsも全く同じです。「ここまでがこのSkillsの担当。それ以外は別のSkillsに」と書くだけで、Claudeの判断精度が劇的に上がります。
動かない書き方:
範囲を書かない → ユーザーが範囲外のことを頼む → Claudeが無理に実行して失敗する
動く書き方:
「やらないこと」をOut of Scopeとしてリストにします。
markdown
■ Out of Scope
- 広告クリエイティブの制作
→ design-request skillを使ってください
- 予算の承認・決裁
→ 部門長に直接確認してください
- 競合の価格調査
→ competitor-check skillを使ってください
これはAnthropicの公式skillでも使われているパターンです。
ユーザーがSkillsの範囲外を依頼したとき、Claudeは無理に実行するのではなく、適切な別のSkillsを提案するか確認を求めます。失敗を未然に防げるんです。
データも明確です。このパターンは高品質Skillsの70%に出現していて、低品質Skillsにはほぼ出現していません。 マジでここが分かれ目です。
最後は資料の「長さ」です。意外と見落とされがちですが、効果は大きい法則です。
長いSkillsはClaudeの集中力を奪う
■ 法則6: 500行以内に収める
経営会議に出す資料を考えてみてください。
1枚のエグゼクティブサマリーと、詳細分析の別紙。全部を1つのドキュメントにまとめたら、誰も読みません。要点は1枚目、深掘りしたい人だけ別紙を見る。この構造が資料を機能させます。
SKILL.mdも全く同じです。入り口は1枚サマリー、詳細は参照ファイルに逃がす。
Skillsが呼び出されると、その全文がClaudeのコンテキストに読み込まれます。
2,000行のSkillsだと、何もする前に5,000トークン以上を消費します。しかも、Skillsが長いほど途中でClaudeがフォーカスを失い、末尾の指示を無視する確率が上がります。
元記事によると、Anthropic公式のSkillsにはfrontend-design、code-review、security-guidanceがあり、すべて300行以内で設計されているとのことです。コミュニティで人気のSkillsも同様にコンパクトに作られています。
実際に運用してみると、入り口のSKILL.mdは50行以内が理想です。200行でも正直長い。SKILL.mdは薄く保って、詳細は全て参照ファイルに逃がす。Claudeは必要なときだけ参照ファイルを読みにいくので、常時コンテキストを圧迫しません。元記事の「500行以内」→公式の「300行以内」、そして実際に回してみた感覚では「入り口50行以内」。この段階的な絞り込みが、運用する中で見えてきた実感です。
では、長くなったらどうするか？分割しましょう。
markdown
skills/weekly-report/
├── SKILL.md              ← 50行以内（常に読み込まれる）
├── KPI_DEFINITIONS.md    ← 必要なときだけ
├── PAST_EXAMPLES.md      ← 必要なときだけ
└── TEMPLATES.md          ← 必要なときだけ
SKILL.mdにはメインの指示だけを書き、詳細パターンやリファレンスは別ファイルに分けます。Claudeはタスクが実際に必要とするときだけ補助ファイルを読み込みます。
6つの法則が出そろいました。
どの法則も、突き詰めれば「自分の仕事を構造化してClaudeに渡す」設計です。冒頭でオンボーディング資料に例えましたが、6つ通して見ると、まさに仕事の受け渡し方を整理するプロセスそのものだと感じるのではないでしょうか。
実際に全部を適用するとどうなるか、見てみましょう。
6つの法則は50行に収まる
6つの法則をバラバラに意識するのは大変に感じるかもしれません。でも実際には、全部を1つのSkillsに入れても50行で収まります。
実際にマーケの週次レポートを例に、6つの法則が入ったSkillsを作ってみましょう。
まず、フロントマターの説明文です。
markdown
---
name: weekly-report
description: "キャンペーンの週次レポートを生成する。
'レポート', '週次', '振り返り', 'KPI'
というキーワード、または月曜の定例MTG前に使う"
---
「何を」「いつ」「キーワード」が入っています。法則1そのものです。
次に、本文の指示部分です。
markdown
過去1週間のキャンペーンデータを集計する。
以下の形式でレポートを生成:
■ KPIサマリー
- CPA / ROAS / CVR を表形式で
■ 前週比と要因分析
- 各指標の増減理由を1行ずつ
■ 来週アクション
- 改善施策を3点、優先度付きで
制約:
- 数値は小数点1桁まで
- 前週比は矢印 (↑↓→) で表記
命令形の指示が法則2。出力形式の指定が法則3。「過去1週間のデータを集計する」で既存データを先に確認する「まず読め」ステップが法則4。
そして範囲外の定義です。
markdown
■ Out of Scope
- 予算の再配分
- クリエイティブの制作
- 競合分析
やらないことの明記が法則5。全体で50行未満に収まっているのが法則6。
営業なら商談フォローアップ、人事なら面接フィードバック整理など、繰り返す業務がある限りSkills化できます。
大事なのは、6つの法則を「全部暗記すること」ではありません。
このくらいのコンパクトなSkillsに全部が自然に入るということです。
1つ書いてみれば、感覚はすぐつかめます。
良い例を見ました。では逆に、「壊れているSkills」はどこが壊れているのか見てみましょう。
動かないSkillsには共通の壊れ方がある
100個の分析から見えた失敗パターンを3つのカテゴリに整理します。自分のSkillsと照らし合わせてみてください。
■ 説明文の失敗
50文字未満で短すぎる
視点がバラバラ（「このSkillsはレポートを作ります...」と「レポート生成。月曜朝の定例前に使う...」が混在している）
トリガーキーワードが入っていない
どんな場面で使うかの文脈がない
■ 内容の失敗
会話調で書いている
出力形式が指定されていない
「まず読め」ステップがない
範囲外が書かれていない
1,000行を超えている
■ 設計の失敗
1つのSkillsで5つのことをやろうとしている
特定プロジェクトにハードコードされている
Skills自体のバージョン管理がない
初回作成後に一度も更新していない
上の項目で3つ以上チェックが付いたら、そのSkillsは呼び出されていないか、不良出力を出している可能性が高いです。
自分のSkillsフォルダを開いて、一番よく使うものから確認してみてください。
当てはまるものがあっても大丈夫です。次のチェックリストで今日から修正できます。
7項目のチェックで今日から直せる
まず、自分のSkillsの中で一番気になるものを1つ開いてください。
そして、以下の7項目を順番にチェックします。
トリガーキーワードが3つ以上入っているか？ → 30秒で確認、1分で修正できます
最初の250文字にユースケースが書かれているか？ → 30秒で確認、1分で書き足せます
命令形と番号付きステップで書かれているか？ → 30秒で確認、5分で書き直せます
出力形式が明示されているか？ → 30秒で確認、3分で追加できます
既存ファイルを先に読む指示が入っているか？ → 30秒で確認、1分で追加できます
やらないことリストがあるか？ → 30秒で確認、2分で追加できます
500行以内に収まっているか？ → 30秒で確認できます。長ければ分割を検討
合計15分もあれば、1つのSkillsを完全にチェックして修正できます。
修正したら、手動で呼び出さずにClaudeにそのSkillsが対応すべきタスクを依頼してみてください。
Claudeが正しいSkillsを自動で選べば、descriptionが機能しています。選ばなければ、descriptionにトリガーキーワードや文脈を追加しましょう。
そしてここが面白いところなんですが、ダメなSkillsはClaudeのコンテキストを無駄に消費して、全体のパフォーマンスを下げます。
でも良いSkillsは逆方向に複利で効きます。1つ良くなると、Claude全体のSkill選択精度が上がるんです。
Skillsは「コレクション」ではなく「システム」になります。
6ヶ月間これを続けた人のskillsフォルダと、始めたばかりの人のskillsフォルダはマジで別物です。
同じClaude Codeを使っているのに、出力の質がまるで違う。その差を生むのが、この6つの設計法則です。
ここまで6つの法則を見てきて、お気づきの方もいるかもしれません。
どのパターンも、プログラミングの知識は要りません。
descriptionを丁寧に書く。指示を明確にする。出力形式を決める。既存のやり方を先に確認させる。範囲を定義する。コンパクトにまとめる。
新しいメンバーへのオンボーディング資料を作るのと、やっていることは全く同じです。
これは全部、「自分の仕事をどう構造化するか」「AIに何を、いつ、どの形式で伝えるか」という判断の話です。

Claude Code に仕事を任せて、いいリズムで進み始めた矢先に「上限に達しました」と表示されて手が止まる。
そんな経験はありませんか？ 思い切って Max プランに上げたのに、月末を待たずに枠を使い切ってしまう月がある。料金で解決したはずなのに、また同じところで止まる。
うまく使えていない原因は、プランの安さよりも使い方にあることがほとんどです。もっと言えば、一番賢いモデルに、どんな作業でも、毎回いっぱいの力で取り組ませている。これは、時給の高い優秀な人に、コピー取りからデータ整理まで全部頼んでいるようなものです。
Claude Code は、賢さも料金も違う複数の社員を雇っているのと同じです。誰にどの仕事を、どれくらいの力で任せるか。その割り振りこそが、上限に振り回されないための一番の近道になります。海外では、Claude の利用枠を上手に使う習慣を10個にまとめた記事があります
 元記事はこちら：
https://x.com/0x_kaize/status/2038286026284667239)。

ここではその内容を土台に、Claude Code を使うビジネスの現場向けに置き換え、2026年5月時点の最新情報 (新しいモデル、緩和された上限、6月15日からの課金の変更) に更新したうえで、プランを上げずに上限に当たる回数を減らす判断基準を、作業別の具体例つきで解説します。
①なぜ「上限に達しました」と言われるのか
まず、何を使い切っているのかを知っておきましょう。Claude が数えているのは、やり取りした回数ではありません。トークンと呼ばれる、文字のかたまりの量です。ざっくり言えば、長い文章をやり取りするほど多く消費します。
ここで見落としやすいのが、会話が長くなるほど1回のやり取りが高くつくという点です。Claude は新しい返事をするたびに、それまでの会話をはじめから全部読み直しています。会話が10往復、20往復と伸びると、毎回その全部をもう一度読むことになります。元記事の試算では、30往復目のやり取りは1往復目のおよそ31倍のコストがかかり、長い会話では消費の大半が「過去の読み直し」に消えていました。
つまり、同じ作業でも、短く区切って的確に頼むほど枠は長持ちします。原因が「回数」ではなく「読み直す量」だと分かれば、打ち手も見えてきます。会話をだらだら続けず、要点を絞る。それだけで消費はかなり変わります。
ただ、その工夫に入る前に、知っておくべき朗報があります。2026年に入って、上限そのものが大きく変わりました。
②上限は2026年に入って大きく緩和された
節約の話を始める前に、現状を正しく押さえておきます。少し前まで出回っていた節約術の中には、今ではもう必要のないものが混じっているからです。
2026年5月6日、Anthropic は Claude Code の利用枠を大きく広げました。具体的には、5時間ごとの利用枠が Pro、Max、Team の各プランで2倍になりました。さらに、混み合う時間帯に枠の減りが早くなる「ピーク時間の制限」が、Claude Code の Pro と Max で撤廃されました。週単位の上限も、2026年7月13日までは一時的に50%上乗せされています。背景には、SpaceX との大規模な計算資源の契約があります (メンフィスのデータセンターで、30万キロワットを超える電力と、22万を超える GPU を確保したと発表されました)。
ここで大事なのは、元記事にもあった「混み合う時間を避けて作業しよう」という工夫が、Claude Code についてはもう不要になったことです。少し前は平日の日中に枠が早く減りましたが、その制限は取り払われました。時間帯を気にして作業を後ろ倒しにする必要はありません。
一方で、5時間ごとに枠が回復していく仕組みそのものは今も同じです。朝にまとめて使い切るより、午前と午後に分けて使うほうが、回復した枠を無駄なく使えます。
では、時間帯ではなく「使い方」で一番効くのは何か。最も大きな節約は、モデルの選び方にあります。
③最大の節約はモデルの使い分け
Claude Code には、Opus、Sonnet、Haiku という3種類のモデルがあります。これを、賢さも単価も違う3人の社員だと考えてみましょう。
Opus は最難関の仕事だけを任せるスペシャリストです。最も賢く、最も単価が高い。2026年5月28日に最新版の Opus 4.8 が登場し、料金は100万トークンあたり入力5ドル・出力25ドルのまま、急ぎ用の高速モードが従来の3倍安くなりました。Sonnet は何でもこなす主力で、料金は Opus のおよそ6割 (入力3ドル・出力15ドル)。Haiku はスピード重視の実務担当で、3つの中で最も安く済みます (入力1ドル・出力5ドル)。
一番もったいないのが、すべての仕事を Opus に任せてしまうことです。賢いから安心という理由で一番上のモデルを選びがちですが、それは時給の高いエースに、データ整理から議事録の清書まで全部やらせているのと同じです。経営の感覚で考えれば、すぐにもったいないと分かるはずです。
3人の社員のイメージ
Haiku はスピードと安さが持ち味の実務担当。Sonnet は標準でおすすめの主力で、Claude Code の作業のおよそ9割はこれで十分こなせます。Opus は本当に難しいときだけ呼ぶスペシャリスト、という役割分担です。
作業別の目安
Haiku が向くのは、表記ゆれの一括修正、データの整形、長い文章の要約、定型メールの下書き、ファイルの中身の確認といった、頭をそこまで使わない手作業です。Sonnet が向くのは、資料のドラフト作成、調べた内容を表にまとめる、いつもの修正や実装、議事録から論点を抜き出すといった、日々の実務の大半です。Opus に回したいのは、入り組んだ問題の原因究明、大量の資料を横断した分析、込み入った設計や戦略の壁打ちなど、本当に頭をひねる仕事だけです。
切り替え方と opusplan という裏技
切り替えは Claude Code で /model と打つだけです。/model sonnet、/model haiku のように指定でき、それまでの会話の流れや読み込んだ資料は引き継がれたまま、次の返事を担当するモデルだけが変わります。便利なのが opusplan という設定です。これを選ぶと、計画を立てる場面では Opus がじっくり考え、実際に手を動かす場面では自動で Sonnet に切り替わります。頭の使いどころは Opus、量をこなすところは Sonnet、と勝手に振り分けてくれます。
Opus を全部に使うのをやめて Sonnet を主力にするだけで、消費はかなり下がります。作業によっては、同じ内容でも入力と出力それぞれ4割ほど、使い方しだいでは6割前後まで減らせます。これは単なる節約のテクニックである前に、どの仕事を誰に任せるかという判断そのものです。
モデルという「誰に任せるか」を決めたら、次は「どれくらいの力で取り組んでもらうか」です。それが effort です。
④次に効くのは effort の使い分け
Claude Code には effort、日本語で言えば努力レベルという設定があります。AI にどれだけ力を入れて考えてもらうかの段階で、low、medium、high、xhigh、max の5段階があります。
初期設定は high で、ふだんの作業はこれでほぼ足ります。ポイントは、これを下げると AI の動きが簡潔になり、消費が減ることです。Anthropic も、努力レベルを下げるほど応答が速くなり、消費する枠が少なくなると説明しています。逆に max は、本当に難しい問題のためのものです。ふだんから max にしていると、簡単な作業にまで延々と考え込んでしまい、コストばかりかさみます。
5段階の意味
low は単純で急ぎの作業向け。medium はコストを抑えたいとき。high は標準で、初期設定もこれです。xhigh は込み入った調査や長時間の作業向けで、最新の Opus 4.8 と Opus 4.7 だけで使えます。max は本当に難しい一発勝負のときだけ、という位置づけです。
作業別の目安
定型の修正や整形、要約のような作業なら low か medium で十分です。ふだんの調査、資料づくり、実装は high。長い時間をかけた深い調査や、まとめて任せる作業には xhigh。そして、どうしても難しい判断や設計のときだけ max を使います。max は普段づかいするものではありません。
設定のしかた
/effort と打つとスライダーが開き、レベルを選べます。/effort medium のように直接指定したり、/effort auto で標準に戻したりもできます。
全部を max にする必要はまったくありません。同じモデルでも、作業の重さに合わせて力の入れ具合を変えるだけで、枠の減りが目に見えて遅くなります。これもモデル選びと同じで、AI にどう働いてもらうかの判断です。モデルと努力レベル、この2つの大きなつまみを覚えたら、あとは毎日の小さな習慣が効いてきます。
⑤日々の習慣で差がつく節約術
ここからは、毎日の操作でできる工夫です。どれも「AI に毎回ゼロから読み直させない」ための工夫で、塵も積もれば枠が大きく変わります。
会話を区切る
作業の区切りがいいところで /compact と打つと、それまでの会話を要約して軽くしてくれます。会話が重くなって動きが鈍る前に、こまめに区切るのがコツです。まったく別の作業に移るときは /clear で会話をまっさらに戻します。前の用件を引きずったまま続けると、関係のない履歴まで毎回読み直すことになり、もったいないからです。
今の状態を確認する
/context と打つと、いま何が会話の容量を占めているのかが一覧で見えます。読み込んだファイルや過去のやり取りが膨らんでいないか、ときどき確認すると無駄に気づけます。
依頼を具体的にする
「全体をいい感じに直して」より「この部分をこう直して」と頼むほうが、AI が読む範囲が狭くなり、消費も抑えられます。範囲を絞るほど安く、答えも的確になります。
CLAUDE.md に常設の指示を置く
毎回伝えている前提 (自分の役割、文章の好み、進め方のルール) は、CLAUDE.md という1枚のファイルに書いておけます。一度書けば毎回読み込んでくれるので、説明のし直しが要りません。長くなりすぎると逆効果なので、200行以内を目安にします。
※ 200行は"最悪この長さ"の認識でいるとOKです。よく言われるのは50行前後に押さえてより詳しい情報はreferencesのようにサブディレクトリを用意してそこで詳しい情報のmdファイルへのリンクを貼る構造です。私も普段この構造でCLAUDE.mdやSKILL.mdを構築しています。
繰り返す資料はキャッシュが効く
同じ資料を何度も使う場合、Claude Code は自動でキャッシュ (一時的な記憶) を使います。一度読み込んだ内容は、次からはおよそ1割のコストで読み直せます。よく使う資料ほど、この仕組みが効いてきます。
ここまでが日常の最適化です。最後に、2026年6月15日から始まる課金の変更で、自分が影響を受けるのかどうかを確認しておきましょう。
⑥6月15日の課金変更 あなたは気をつけるべきか
2026年6月15日から、Claude の課金の仕組みが一部変わります。ニュースで見て不安になった方もいるかもしれませんが、先に言うと、大多数の人には関係ありません。
変わるのは、AI に自動で連続作業をさせる使い方だけです。具体的には、画面で対話せずに命令を一括で自動実行する使い方 (ターミナルで claude -p という形で動かすもの) や、外部のツールや自動化の仕組みと Claude をつないでいる場合です。こうした使い方が、6月15日からはこれまでの契約の枠とは別の、専用の枠で扱われるようになります。
影響を受けない人 (大多数)
画面で対話しながら Claude Code を使っている人、ふだんチャットで使っている人は、これまでと何も変わりません。いつも通りです。
気をつけたほうがいい人
ターミナルで自動実行 (claude -p) を回している、外部ツールや自動化の仕組みと連携させている、開発した仕組みに組み込んでいる、といった使い方をしている人です。
その人がやること
この専用の枠は、Pro で月20ドル相当、Max 5x で100ドル相当、Max 20x で200ドル相当が毎月割り当てられます。月ごとにリセットされ、繰り越しはできません。足りなくなりそうなら、次に説明する追加利用を設定しておきます。
自動化を組んでいない大多数の人にとっては、安心して構いません。これは、一部の重い自動利用が全体の枠を圧迫していた状況を整理するための変更で、対話で使う人の枠はむしろ守られる方向です。自分がどちらに当てはまるかを見極めて、必要な人だけ手を打てば十分です。
では、それでも枠が足りなくなりそうなときの備えを見ておきます。
⑦それでも足りない時の備え
使い分けを徹底しても、大事な作業の途中で枠が尽きそうになることはあります。そのための保険が、追加利用 (設定画面では Overage と表示されます) です。
設定画面の Usage から有効にしておくと、上限に達しても作業が止まらず、使った分だけ追加で課金される形に切り替わります。標準の利用料金で、使った分だけです。使いすぎが不安なら、月ごとの上限金額を決めておけますし、1日あたりの上限も2000ドルと定められています。これは節約というより、いざというときに作業を失わないための備えです。
プラン選びも、初期設定のモデルで考えると分かりやすくなります。Pro は標準が Sonnet、Max は標準が Opus です。ふだん Opus を多用するなら Max、Sonnet 中心で足りるなら Pro でも十分なことがあります。自分の使い方に合わせて選ぶことが、結局は一番の節約になります。
最後に、ここまでをまとめておきます。
⑧まとめ
節約というと細かい我慢のように聞こえるかもしれません。けれど実際にやっているのは、AI という人材をどう使い分けるかという判断です。一番賢いモデルに全部・全力でやらせるのをやめて、仕事の重さで割り振る。それだけで、同じ料金のまま Claude Code はずっと長く働いてくれます。
もし1つだけ試すなら、次に Claude Code を開いたとき、モデルを1段下げる (Opus を Sonnet に) か、努力レベルを1段下げて、同じ作業をやってみてください。たいていの作業は、それで何も困らないことに気づくはずです。
Claude が数えているのは回数ではなく文字の量。会話が伸びるほど読み直しで高くつく
2026年5月に上限は緩和された。混み合う時間を避けるという昔の工夫は、Claude Code ではもう不要
一番効くのはモデルの使い分け。9割は Sonnet で足り、本当に難しいときだけ Opus に回す。opusplan も便利
次に効くのが努力レベル (effort)。初期設定の high で十分で、max は本当に難しいときだけ
会話は区切り (/compact、別件は /clear)、依頼は具体的に、CLAUDE.md で毎回の説明を省く

■ 𝟭. `/simplify` が `/code-review` に進化した（v2.1.147・5/21）
まず最初がこれ。普段づかいのコマンドがまるっと別物になったので、知らずに使うと一番「あれ？」となるやつです。
これまであった `/simplify` コマンドが、今週から名前も中身もガラッと変わって `/code-review` になりました。
旧 `/simplify` は「コードを片付けて直す（cleanup-and-fix）」挙動でしたが、この古い挙動は完全に削除されています。新しい `/code-review` は、コードを勝手にいじるのではなく、正しさ（correctness）のバグを指摘するレビュー役に振り切りました。
しかも、レビューの深さを自分で選べます。
```
/code-review high
```
のように effort レベルを指定すると、その強度でレビューしてくれます。軽く見てほしいときは低く、本気で粗探ししてほしいときは高く、と使い分けられるわけです。
さらに強力なのが `--comment` オプション。
```
/code-review --comment
```
これを付けると、見つけた指摘をGitHub の PR に inline コメントとして直接投稿できます。つまり「Claude にレビューさせて、その結果をそのままチームのプルリクに貼る」という流れが、コマンド一発で完結します。
𝗕𝗲𝗳𝗼𝗿𝗲：`/simplify` で何となくコードを整えてもらう（けど勝手に直される）
𝗔𝗳𝘁𝗲𝗿：`/code-review high --comment` で、正しさのバグだけを指摘 → PRにコメントとして残す
ざっくり言うと「AIに掃除させる」から「AIにレビューさせる」への方針転換です。これまで `/simplify` を使っていた人は、もう同じ名前では呼び出せません。`/code-review` に読み替えてください。

■ 𝟭. `/simplify` が `/code-review` に進化した（v2.1.147・5/21）
まず最初がこれ。普段づかいのコマンドがまるっと別物になったので、知らずに使うと一番「あれ？」となるやつです。
これまであった `/simplify` コマンドが、今週から名前も中身もガラッと変わって `/code-review` になりました。
旧 `/simplify` は「コードを片付けて直す（cleanup-and-fix）」挙動でしたが、この古い挙動は完全に削除されています。新しい `/code-review` は、コードを勝手にいじるのではなく、正しさ（correctness）のバグを指摘するレビュー役に振り切りました。
しかも、レビューの深さを自分で選べます。
```
/code-review high
```
のように effort レベルを指定すると、その強度でレビューしてくれます。軽く見てほしいときは低く、本気で粗探ししてほしいときは高く、と使い分けられるわけです。
さらに強力なのが `--comment` オプション。
```
/code-review --comment
```
これを付けると、見つけた指摘をGitHub の PR に inline コメントとして直接投稿できます。つまり「Claude にレビューさせて、その結果をそのままチームのプルリクに貼る」という流れが、コマンド一発で完結します。
𝗕𝗲𝗳𝗼𝗿𝗲：`/simplify` で何となくコードを整えてもらう（けど勝手に直される）
𝗔𝗳𝘁𝗲𝗿：`/code-review high --comment` で、正しさのバグだけを指摘 → PRにコメントとして残す
ざっくり言うと「AIに掃除させる」から「AIにレビューさせる」への方針転換です。これまで `/simplify` を使っていた人は、もう同じ名前では呼び出せません。`/code-review` に読み替えてください。

でも、こういう情報って、専門用語まみれの発表記事を読んでも、初心者にはさっぱり頭に入ってこないですよね。

そこでこの記事では、AIをこれから使いこなしたい初心者の方に向けて、「Claude Opus 4.8」「Claude Code」「Claude Design」という3つのキーワードを、できるだけやさしい言葉で、しかも「あなたが今日から何をすればいいか」までセットで解説していきます。
専門知識はいりません。

第1章 そもそもこの3つ、何がすごいの？
最初に、いちばん大事なところだけ先にまとめます。
細かい話は後の章でゆっくりやるので、ここでは「ざっくりした地図」を頭に入れてください。
Claude Opus 4.8は「いまいちばん賢いClaude」
まず主役の「Claude Opus 4.8」から。
これは、AIアシスタント「Claude（クロード）」シリーズの中で、いちばん賢い最上位モデルの最新版です。
「モデル」というのは、AIの頭脳そのもの、というイメージで大丈夫です。
同じClaudeでも、軽くて速い頭脳、バランス型の頭脳、めちゃくちゃ賢い頭脳、といくつか種類があって、その「めちゃくちゃ賢い枠」の最新版がOpus 4.8なんですね。
今回のバージョンは、2026年5月28日に公開されました。
ポイントは大きく2つあります。
1つ目は、値段がひとつ前のバージョン（Opus 4.7）と同じまま、賢さだけが上がったこと。
プログラミング、複雑な調べ物、長い文章の読み込みといった、頭を使う作業がさらに得意になりました。
2つ目は、「正直さ（honesty）」が改善されたこと。
これは、AIがわからないことを「わかったフリ」で答えてしまう、いわゆる嘘っぽい回答が減った、という意味です。
初心者にとっては、ここがかなり大事なんですよね。
AIの答えを鵜呑みにして失敗する、というリスクが少し下がったわけですから。
具体的に「賢い」ってどういうこと？と思いますよね。
たとえば、長いPDF資料をまるごと読ませて要点を出させる、こみ入った相談に筋道立てて答えてもらう、あいまいな指示から意図をくみ取ってもらう。
こういう「人間がやると頭が疲れる作業」を、Opus 4.8はかなり安定してこなしてくれます。
逆に言うと、軽い雑談やちょっとした調べ物なら、ここまで賢いモデルじゃなくても十分なんですよね。
だからこそ、次の章で出てくる「力の入れ具合を選ぶ」という発想が効いてくるわけです。
Claude CodeとClaude Designは「賢いClaudeの使い道」
次に、残りの2つです。
「Claude Code」と「Claude Design」は、モデル（頭脳）ではなく、その賢い頭脳を「どこで、どう働かせるか」という道具・作業場のことだと思ってください。
Claude Codeは、ものすごくざっくり言うと「AIにパソコンの中の作業を任せるための仕組み」です。
もともとはプログラマー向けの道具で、ファイルを読んだり、書き換えたり、動作チェックまで自分でやってくれます。
今回のアップデートで「dynamic workflows（ダイナミック・ワークフロー）」という新機能が加わって、ひとつの指示から、たくさんの小さな作業を同時並行で進められるようになりました。
一方のClaude Designは、2026年4月に始まった「会話しながらデザインや資料を作る道具」です。
画面の右側にキャンバス、左側にチャットがあって、「こういう資料を作って」と話しかけると、スライドやチラシ、Webページのデザイン案を出してくれます。
ここまでをひとことでまとめると、こうなります。
賢く考えるのがOpus 4.8、パソコン作業を任せるのがClaude Code、見た目や資料を作るのがClaude Design。
この3つの役割分担さえ覚えておけば、もう迷子にはなりません。

サッと答えがほしい簡単な質問なら、力を抜いた設定のまま。

コピペで使える「お願いの型」とAI活用の注意点
AIに何かを頼むとき、多くの初心者がやりがちなのが、「いい感じにして」みたいなフワッとしたお願いです。
これ、人間相手でも難しいですよね。
AIも同じで、伝え方次第で答えの質がガラッと変わります。
そこで、コピペして使える「お願いの型」を用意しました。
あなたは{役割}です。

目的：{達成したいこと}

前提：{読む人や状況}

条件：
- わからないことは「わからない」と書く
- むずかしい言葉には説明をつける

出力：{箇条書き／表／文章など、ほしい形}
たとえば「あなたはやさしい先生です」「目的は、ふるさと納税のしくみを理解すること」みたいに、カッコの中を自分の言葉で埋めるだけ。
これだけで、回答の精度が見違えるほど上がります。
そしてここから、すごく大事な話をします。
AIをSNS投稿やブログ、note記事づくりに使うとき、絶対にやってはいけないことがひとつあるんです。
それは、「AIに丸投げして、出てきた文章をそのままコピペで投稿する」こと。
正直に言うと、これがいちばんもったいない使い方なんですよね。
理由は2つあります。
1つ目は、AIが書いたままの文章って、どこか他人事で、温度が低いから。
読む人は意外と敏感で、「これAIっぽいな」という空気をすぐ感じ取ります。
2つ目は、最近のSNSは「AIが量産したような中身の薄い投稿」を見抜いて、表示を抑える方向に動いていること。
つまり、丸投げ投稿はバレるし、伸びにくいんです。
じゃあどう使うのが正解かというと、役割分担なんです。
AIには、下書き・たたき台・壁打ち相手をやってもらう。
そして、自分の体験談、本音、ちょっとした失敗、温度のある言葉を、最後に人間が足す。
この「AIが7割、人間が3割の仕上げ」をやるだけで、文章は一気にあなたのものになります。
AIは、あなたの代わりに発信する道具じゃなくて、あなたの発信を加速させる相棒。
この感覚を持てるかどうかが、これからのAI活用で、いちばん差がつくポイントだと思います。
フワッとした依頼と、型を使った依頼はこんなに違う
百聞は一見にしかずなので、同じ用件を「フワッと頼んだ場合」と「型を使って頼んだ場合」で比べてみますね。
たとえば、自己紹介文を作りたいとします。
フワッとした依頼は、こんな感じです。
「自己紹介を書いて」
これだと、AIは相手のことを何も知らないので、当たりさわりのない、ありきたりな文章しか返せません。
一方、型を使うとこうなります。
「あなたはプロのライターです。目的は、私のXのプロフィール文を作ることです。私は30代の会社員で、子育てしながら副業でハンドメイド雑貨を売っています。やわらかい雰囲気で、140文字以内でお願いします」
どうでしょう。
後者なら、あなたらしさの出た、すぐ使えるプロフィール文が返ってきます。
ポイントは、「誰が」「何のために」「どんな状況で」「どんな形で」ほしいのかを、ひとことずつ足してあげること。
最初は面倒に感じるかもしれませんが、慣れると数秒でできるようになります。
この「ひと手間」こそが、AIをうまく使える人と、そうでない人の分かれ道なんです。
こんなときに使える、初心者の活用シーン集
「賢いのはわかったけど、自分の生活のどこで使えるの？」
ここがイメージできると、AIは一気に身近になります。
初心者でもすぐ試せるシーンを、コピペで使える例文つきで5つ挙げますね。
1つ目は、仕事の文章づくりです。

気まずくて筆が止まる文章ほど、AIは落ち着いて書いてくれます。
2つ目は、長い情報の要約です。
例文は「この記事を、忙しい人向けに3行でまとめて」。
ニュースや会議資料を読む時間がないとき、ものすごく助かります。
3つ目は、学び直しです。

「いまさら聞けない」ことを、何度でも気がねなく聞けるのが、AIの最高なところなんですよね。
4つ目は、発信のネタ出しです。
例文は「30代向けの節約アイデアを、SNSの投稿ネタとして10個出して」。
ただ、ここで第4章の話を思い出してください。
出てきたネタをそのまま投稿するのではなく、自分の体験をひとつ混ぜる。
それだけで、どこかで見たような投稿が、あなたにしか書けない投稿に変わります。
5つ目は、暮らしの相談です。
例文は「冷蔵庫にある卵・キャベツ・豚肉で作れる夕飯を、3つ提案して」。
献立、旅行プラン、お金のやりくり。
日々の小さな「どうしよう」を、気軽に投げてみてください。
こうして並べてみると、特別な知識がなくても使える場面って、想像以上に多いんです。
まずはこの中から、「あ、これ今日やってみたいかも」と思ったものを、ひとつだけ試してみてください。
第5章 失敗しないコツと「3つの道具の連携」
道具の使い方がわかってきたら、次は「失敗しないコツ」です。
ここを知っているかどうかで、AIに対する満足度がまるで変わってきます。
コツは「いきなり完璧を狙わない」こと
初心者がAIにガッカリするパターンって、だいたい決まっているんです。
それは、最初の一発で完璧な答えを期待して、思ったものが出てこなくて「やっぱり使えないじゃん」となるケース。
でも、ちょっと待ってください。
これ、AIの実力不足じゃなくて、使い方の問題なんですよね。
うまい人は、いきなり完成品を狙いません。
まず「こういうものを作りたいんだけど、どんな構成がいいと思う？」と相談する。
骨組みに納得してから、「じゃあその構成で書いてみて」と本文をお願いする。
そして出てきたものに、「ここはもっとやさしく」「この例は別のものに変えて」と注文をつけて、何度か直してもらう。
この「相談 → たたき台 → 修正」を2、3回くり返すだけで、最終的な質は別物になります。
もうひとつ、プロもよく使う裏ワザを教えますね。
それは、AIに自分の回答をチェックさせること。
「いまの回答の弱点を3つ挙げて」とか「初心者がつまずきそうな点はどこ？」と聞くんです。
AIは自分のアラを意外と素直に教えてくれるので、それをもとにもう一度直させると、ぐっと完成度が上がります。
完璧主義を手放して、一緒に育てていく感覚。
これが、AIとうまく付き合う最大のコツです。
たとえば、記事のタイトルを一緒に育ててみる
イメージがわくように、実際のやり取りを再現してみますね。
note記事のタイトルを考える、という場面でいきましょう。
最初に、ためしにこう頼みます。
「節約の記事のタイトルを5個出して」
すると、たしかに5個出てきますが、どこかで見たような、ありきたりなものばかりです。

押さえとくべき挙動は3つ。
①ファイルが更新されると、NotebookLM側のソースも更新される

③元ファイルがDriveから削除されると、NotebookLMからも除去される
データのライフサイクルが元ファイルに完全紐づく設計。
この3つの挙動見ると、Googleは「NotebookLMはDriveの拡張」っていう位置づけを明確にしたんだなって伝わってきます。
ロールアウト中なので、まだ動いてなくても焦るな

YouTubeの落とし穴：字幕ないと無理ゲー
YouTubeのURL入れるとき、これマジでハマる人多いんですけど、字幕がない動画は取り込めません。

画像と手書きノート：「使える」けど「保証はない」

特に、手書きノートの認識精度は公式保証なし。なので、運用設計するときは「使えればラッキー」くらいで考えとくのが安全です。
実際に試してみると、印刷物の表や図、PowerPointの図表はかなり高精度で読み取ってくれます。
でも、達筆な手書きメモとか、複雑な数式が混ざった黒板写真みたいなのは、認識精度ガクッと落ちる。
業務で画像をソースにしたい場合は、可能な限り「OCR可能なきれいな画像」にしとくか、事前にテキスト化してから入れるのがベター。

mp4とかmpegの動画ファイルも取り込めるんですが、Googleの公式説明をよく読むと、「音声を文字起こしして保存」って書いてあるんですよ。
つまり、動画の映像部分は基本的に解析されてない。
スライドが映ってる画面解説動画とかでも、NotebookLMが理解してるのは「音声で話されてる内容」だけ。画面に表示されてるテキストや図は読み取れてません。
これ、勘違いしやすいポイントなので絶対覚えといてください。
業務で動画を分析したいなら、可能なら別途スライド資料も並行してアップロードするか、事前に動画から画面キャプチャを取って画像ソースとして入れるのが現実的です。

ページ内に埋め込まれてる画像、動画、JavaScriptで動的表示されるコンテンツ、リンク先のネストされたページは取り込めない。
さらに、ペイウォールの向こう側にあるコンテンツも当然無理。

Flashcards・Quizzesは業務にも使える
教育用途のイメージ強い機能ですが、業務でも使い道あります。
たとえば、新人研修。
会社の業務マニュアルをNotebookLMに入れて、Flashcardsを生成すれば、新人向けの自習用暗記カードが一瞬で作れる。
Quizzesを生成すれば、研修後の理解度テストが自動生成される。
また、自分が新しい業務領域を勉強したいときにも便利。専門書や業界レポートをまとめて入れて、Flashcardsを作れば、隙間時間で重要用語や概念を反復学習できる。
「生成回数」には1日あたりの上限がある
ここ要注意ポイントなんですが、Audio Overview、Video Overview、Reportsなどの生成回数には、プランごとに1日あたりの上限があります。
標準プランだと、Audio Overviewは1日3回、Reportsは1日10回まで。
業務でガッツリ使うなら、後述するPro以上のプランを検討する必要あり。
第5章：Gemini連携で“ヤバい”が加速｜資料倉庫×対話表面の完成形
NotebookLM＝倉庫、Gemini＝表面、の分業がヤバい
2026年1月から段階展開された、GeminiでNotebookLMノートをソース追加できる機能。
これ、最初は「ふーん、連携できるんだ」くらいの認識だったんですけど、実際に使ってみるとマジで便利で、業務の使い方が結構変わります。
具体的にどう動くかというと、Geminiアプリでプロンプト欄の「＋」ボタンから「NotebookLM」を選んで、自分のノートブックを指定。すると、そのノートブックに含まれてる全ソースを根拠として、Geminiが回答するようになります。
これの何が革命的って、「NotebookLMで貯めた根拠資料を、Geminiの会話と生成力で扱える」ようになったこと。
NotebookLMだけだと、回答が比較的「お堅い」というか、ソース引用ベースの真面目な回答になりがち。それはそれで業務に合ってるんですが、たとえば「もっとくだけた言い方で社内向けに書き直して」みたいな柔軟な指示には、Geminiの方が得意なんですよ。
そこで、NotebookLM＝資料倉庫、Gemini＝対話と生成の表面、という分業が成立する。
Geminiの強力機能をNotebookLMの根拠で使える
Geminiには、NotebookLM単独では使えない強力な機能がいくつかあります。
Gem：Geminiのカスタムアシスタント機能。特定の役割を持ったAIアシスタントを永続的に作れる。NotebookLMノートをソース追加したGemを作れば、「うちの会社の営業ナレッジを全部知ってる営業アシスタント」みたいなのが作れる。
Canvas：ドキュメント作成ワークスペース。NotebookLMの資料を根拠にしながら、Canvasで長文ドキュメントを書ける。提案書、報告書、ブログ記事、社内通達、何でも作れる。
Deep Research：Geminiが自律的に大規模リサーチして、レポートを作ってくれる機能。NotebookLMの社内資料を起点に、外部情報を組み合わせた調査ができる。
Veoは動画生成、Guided Learningは学習支援機能。これらもNotebookLMの根拠を使えるようになりました。
つまり、NotebookLMに資料を貯めとけば、Geminiのあらゆる機能でその知識を活用できるようになったわけです。これ、マジで強い。
管理者側の設定で押さえとくべきポイント
業務利用で重要なのが、この連携の管理権限がどこにあるかって点。
Google公式によれば、「Geminiアプリ内でノートブックをソースとして使う機能は、Geminiアプリの設定ではなく、NotebookLMのサービス設定により決まる」となってます。
組織導入では絶対に押さえてほしいんですが、Geminiを社内で開放してても、NotebookLMサービスがOFFになってれば、この連携は成立しない。
逆に、Geminiは閉じててNotebookLMだけ開けてるケースだと、Gemini連携機能自体が使えない。
Workspace管理者は、両方のサービスを連携させる前提で設定する必要があります。
業務での実用シナリオ、具体的に書きます
実際にどう使うか、業務シナリオで考えます。
シナリオ1：営業の提案資料作成
NotebookLMに、過去の成功提案書・商品スペック・FAQ・競合比較資料を全部入れとく。
新規案件が来たら、Geminiでそのノートブックをソース指定して、「この顧客向けに、A社の事例を参考にした提案骨子を作って」と指示。
GeminiがNotebookLM内の根拠資料を引きながら、Canvasで提案ドラフトを書いてくれる。マジで時短。
シナリオ2：社内FAQボットの代替
人事ルール、経費ルール、業務手順書をNotebookLMに入れとく。
社員がGeminiに「育休の取得手続きについて教えて」と聞くと、Geminiが該当ノートブックを根拠に、社内ルール準拠の正確な回答を返す。
しかもDrive自動同期があるので、ルールが更新されたら自動的に最新版で回答される。革命。
シナリオ3：マネージャーの意思決定支援
各部署の週次レポート、KPI、議事録をNotebookLMに集約しとく。
マネージャーがGeminiで「今週、優先的に注意すべき案件はどれ？」と質問すると、Geminiが各レポートを横断して、優先度の高い案件を抽出してくれる。
Gem化で永続AIアシスタントを作る
Gemini連携で一番強力なのが、Gemへの組み込み。
Gemは、Geminiのカスタムアシスタント機能で、特定の役割・知識・性格を持たせたAIを永続的に保存できる。
NotebookLMノートをGemに組み込めば、「特定業務に特化した、社内ナレッジを完全に把握してるアシスタント」が完成する。
たとえば、
「営業1課のナレッジアシスタント」：1課の提案書、商品資料、顧客リストをノートブックにまとめて、Gem化。1課のメンバー全員が、Geminiで簡単にナレッジを引ける。
「カスタマーサポート対応Gem」：FAQ、過去のサポート対応履歴、製品マニュアルをノートブックにまとめて、Gem化。新人サポートメンバーの教育補助にも使える。
「法務チェックGem」：契約テンプレート、過去の修正履歴、社内法務ルールをまとめて、Gem化。営業や事業部からの法務相談の一次受けに使える。
このGem化、業務のスケールメリットがマジでデカいんですよ。

監査ログ：誰がいつ何をしたかの完全な記録。コンプライアンス対応に必須。
これだけ揃うと、大企業の本格導入要件にほぼ全部応えられる。

第7章：知らないと事故る！企業導入のセキュリティ落とし穴まとめ
「気づいたら情報漏洩」を防ぐための前提知識
NotebookLMを業務導入するとき、これを知らないとマジで事故るポイントがいくつかあります。
順番に解説していきます。
Drive取り込みでも「コピー」が作られる罠
これ、見落とされがちなんですけど、Driveから取り込んだファイルでも、NotebookLM側に新しいコピーが作られます。
つまり、元のDriveファイルをアクセス権限で守ってても、NotebookLMに取り込まれた時点で、そのコピーはNotebookLM側の管理下に移る。
これが意味するのは、「Driveのアクセス権限とNotebookLMのアクセス権限は別物」ってこと。
たとえば、機密情報が入ったDriveドキュメントを「部長のみアクセス可能」に設定してても、ある社員がそれを自分のNotebookLMに取り込んでノートブックを共有してしまえば、元の権限設定は意味をなさなくなる。
これ、企業導入の最初に必ず注意喚起すべきポイントです。
ただし、2026年5月のDrive自動同期アップデートで、アクセス権を失うとNotebookLMからも使えなくなるという挙動が入りました。これにより、退職者対応や権限剥奪はカバーされやすくなった。
それでも、過去にコピーされたデータの扱いについては、ノートブックの共有設定をきちんと管理する必要あり。
Workspace DLPが現時点で統合されてない
これも超重要な制約。
Google WorkspaceにはDLP（Data Loss Prevention：データ漏洩防止）機能があって、「クレジットカード番号を含むファイルは外部共有不可」みたいなルールを自動で適用できます。
ところが、標準のNotebookLM（Workspace版含む）には、このDLPがまだ統合されてません。
つまり、DLPルールで守られてるDriveファイルでも、NotebookLMに取り込んでしまえばDLPの保護下から外れる可能性がある。
これに対するGoogleの代替案は、IRM（Information Rights Management）を使って、Driveファイル側でダウンロード・コピー・印刷を制御するというもの。
ただ、これは根本解決じゃないので、機密度が高い情報を扱う企業は、NotebookLM Enterprise（VPC-SCやModel Armorなどの上位統制が使える）への移行を検討すべき。
データレジデンシー（保存場所）の問題
日本企業で特に注意したいのが、データレジデンシー。
標準のNotebookLMは、Workspaceのデータリージョン設定の対象外。つまり、「データを日本国内のサーバーに置く」という設定があっても、NotebookLMのデータは別の場所に保存される可能性があります。
これ、官公庁向け業務や、金融・医療など個人情報保護法の厳格な業界では致命的になることがある。
NotebookLM Enterpriseなら、US / EU / Global multi-regionから選択可能なので、要件に応じて対応できます。
日本国内ホスティングが必須要件なら、現時点では他の選択肢を検討する必要があるかも。
監査ログの公開情報が限定的
企業の情報セキュリティ部門が必ず聞いてくるのが、「監査ログは取れるんですか？」って質問。
Gemini側の監査ログは、Googleが詳細を公開してます。誰がいつ何のプロンプトを投げて、どんな回答が返ったか、ログとして取得できる。
ところが、NotebookLM固有の監査ログについては、Googleが公開してる情報が限定的なんです。
NotebookLM Enterpriseには「Configure usage audit logging」というドキュメントが用意されてるので、企業向けには監査ログを取れる仕組みがあると考えられます。
しかし、Workspace版で「どのレベルの監査ログが取れるか」については、公開ドキュメントだけでは読み切れない。
導入前に、Google Cloud営業に直接確認するか、SaaS監査ログ管理ツール（DatadogやSplunkなど）との統合可否を確認してください。
プライバシー：Workspace版なら学習に使われない
ここはポジティブな話ですが、Workspace版以上のNotebookLMでは、以下が保証されてます。
アップロードされたファイルが人手レビューされない
入力されたプロンプトがモデル学習に使われない
生成された回答がモデル学習に使われない
フィードバック送信時もこれらの保証が適用される
これは、Workspace契約上の「コアサービス」扱いになったことの恩恵。
業務利用するなら、最低限Workspace版以上で運用すべき理由の最大のひとつがこれ。
コンプライアンス認証の対応状況
法令対応やコンプライアンス認証についても、状況を整理しときます。
Google公式情報によると、NotebookLM（Workspace版）は以下に対応してます。
SOC 1 / 2 / 3
ISO 27001 / 27017 / 27018 / 27701 / 9001
BSI C5
そして、HDS / MTCS / OSPAR / ISMAPについては、2026年6月対応予定（ETA June 2026）と記載されてます。
一方、Google Cloud Japanからは、2026年4月にGemini EnterpriseとNotebookLM EnterpriseのISMAP登録完了が発表されています。
つまり、日本の公共調達適合性（ISMAP）については、Cloud版のEnterprise版は既に対応済み、Workspace版は対応待ちという状況。
公共機関や、公共向けサービスを提供してる企業は、Cloud版Enterprise一択になります。
Context-Aware Access（CAA）の活用
セキュリティ統制を強化したい企業向けに、Context-Aware Access（CAA）という機能があります。
これは、「特定のIPアドレスから」「特定のデバイスから」「特定の時間帯のみ」みたいな条件で、NotebookLMへのアクセスを制限できる機能。
たとえば、「社内ネットワークからしかNotebookLMにアクセスできない」「会社支給PCからしか使えない」みたいなルールを設定可能。
これ、リモートワーク時代のセキュリティ統制としてマジで有効。
NotebookLM Enterpriseだとさらに強力で、プライベートUI経路まで構成できるので、パブリックインターネットを通らずに完結します。
エクスポートとデータポータビリティ
最後に、データのエクスポート機能について。
エンドユーザーはGoogle Takeoutを使って、自分のNotebookLMデータを書き出せる。
管理者はData Export機能で、組織全体のNotebookLMデータをエクスポート可能。
これにより、万が一NotebookLMからの移行が必要になっても、データを取り出して別環境に移せる仕組みは確保されてる。
ベンダーロックインを心配する企業にとって、これは安心材料です。
第8章：実務で勝つ！NotebookLM運用プレイブック完全版
ノートブックの設計思想、ここがマジで大事
NotebookLMを業務で使うとき、最初に決めるべきは「ノートブックをどう切るか」。
これ、適当にやると後で痛い目見ます。
基本原則は、「1テーマ・1業務領域・1ノートブック」。
会社全体を1つのノートブックに詰め込もうとすると、ソース数の上限に引っかかるし、AIの回答精度も落ちます。
逆に、細かく切りすぎると、ノートブック数の上限に引っかかる。
おすすめの粒度は、以下のような単位。
営業部門：案件単位、または「顧客×期間」単位。「ABC社対応 2026年Q2」みたいな切り方。
マーケティング：キャンペーン単位、または「製品×期間」単位。「製品X 2026年春キャンペーン」とか。

業務での使い分けは、「ナレッジベース構築・教材作成・音声/動画/スライドへの変換が中心ならNotebookLM」「対話・調査・コーディング支援・幅広い業務支援ならClaude」となります。

Perplexityの強み：
最新Web情報の探索が圧倒的に強い
出典リンクが明確で、信頼性の高い情報源を優先
Enterprise版でSSO、SCIM、audit logs、configurable retention、GDPR/HIPAA、SOC 2 Type II対応
価格はEnterprise $34/seat/月（年契約）
NotebookLMとの違い：
Perplexityは「Web全体を検索して最新情報を持ってくる」のが本質。NotebookLMは「自分のソースに限定して回答する」のが本質。
リサーチ業務なら：最新の業界動向や競合情報を調べるならPerplexity、社内資料や既存ナレッジに基づく回答ならNotebookLM。

Notion AIの強み：

価格はFree $0、Plus $10/seat/月、Business $20/seat/月、Enterprise見積

■ 𝟭. `/simplify` が `/code-review` に進化した（v2.1.147・5/21）
まず最初がこれ。普段づかいのコマンドがまるっと別物になったので、知らずに使うと一番「あれ？」となるやつです。
これまであった `/simplify` コマンドが、今週から名前も中身もガラッと変わって `/code-review` になりました。
旧 `/simplify` は「コードを片付けて直す（cleanup-and-fix）」挙動でしたが、この古い挙動は完全に削除されています。新しい `/code-review` は、コードを勝手にいじるのではなく、正しさ（correctness）のバグを指摘するレビュー役に振り切りました。
しかも、レビューの深さを自分で選べます。
```
/code-review high
```
のように effort レベルを指定すると、その強度でレビューしてくれます。軽く見てほしいときは低く、本気で粗探ししてほしいときは高く、と使い分けられるわけです。
さらに強力なのが `--comment` オプション。
```
/code-review --comment
```
これを付けると、見つけた指摘をGitHub の PR に inline コメントとして直接投稿できます。つまり「Claude にレビューさせて、その結果をそのままチームのプルリクに貼る」という流れが、コマンド一発で完結します。
𝗕𝗲𝗳𝗼𝗿𝗲：`/simplify` で何となくコードを整えてもらう（けど勝手に直される）
𝗔𝗳𝘁𝗲𝗿：`/code-review high --comment` で、正しさのバグだけを指摘 → PRにコメントとして残す
ざっくり言うと「AIに掃除させる」から「AIにレビューさせる」への方針転換です。これまで `/simplify` を使っていた人は、もう同じ名前では呼び出せません。`/code-review` に読み替えてください。
