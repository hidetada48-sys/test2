## グループ1：トークン節約 — モデルとeffortの使い分け

ブックマーク2・8の引用記事が、消費の正体（会話が伸びるほど「過去の読み直し」で高くつく）と、その対処を作業別に解説しています。あなたは既に `/effort high` をCLAUDE.mdルールに持っていますが、この記事は「常時highは逆にもったいない作業がある」と指摘しており、見直しの価値があります。

### 🛠 環境改善・向上
1. **[P1] `/usage` で消費の内訳を確認する**：いま持っているMCPサーバー（jina・playwright・discord・context7・google系）やスキル12個のうち、どれがトークンを食っているかをカテゴリ別に見られる。あなたは常駐MCPが多いので、重いものを特定して `disabledMcpServers` での無効化候補を判断できる（過去にGmail/Calendar MCPを無効化済みなので、その続きの棚卸し）。（元記事：3・9）
2. **[P1] `opusplan` を試す**：計画はOpus・実装はSonnetに自動で振り分ける設定。たとえば「画像処理スクリプトの大量バッチ修正」のような頭を使わない反復作業を Haiku/Sonnet に落とすだけで消費が4〜6割減るという。（元記事：2）
3. **[P2] effortの常時highを作業別に見直す**：CLAUDE.mdには「複雑なタスクの前にeffort high」とあるが、定型整形・要約のような軽作業は low/medium で十分。「重い作業の前だけ上げる」運用に寄せると枠の減りが目に見えて遅くなる。（元記事：2）

### 📚 学習
4. **[P2] 6月15日の課金変更の対象者を理解する**：`claude -p` の自動実行や外部ツール連携が「専用枠」に分離される。あなたのブックマーク週次収集はsystemdタイマー＋スクリプト運用なので、自分が対象かどうかを一度公式で確認しておくと安心。（元記事：2）

## グループ2：Claude Code 新機能 — エージェントの自走

「test everything で4エージェントが自動テスト」「Managed Agents」「/ultracode の Dynamic Workflows」「Memory Files」と、人が手順を指示せずゴールだけ渡して走らせる流れが共通テーマでした。

### 🛠 環境改善・向上
5. **[P1] `/memory` で公式auto-memoryを管理する**：記事は「追加予定の新機能」と書いていたが、公式確認の結果、Claude Codeでは**すでに正式機能として稼働中**だった（公式: code.claude.com/docs/en/memory）。あなたのMEMORY.md（`~/.claude/projects/<project>/memory/`）がまさにこの公式機能そのもの。`/memory` コマンドで一覧・編集・オンオフ切替ができる。MEMORY.mdは起動時に**先頭200行（または25KB）まで**しか読まれないので、索引は薄く・詳細はトピックファイルに、という運用が正解。（元記事：10／公式裏取り済み）
6. **[P2] Dynamic Workflows を小さく試す**：公式確認の結果、**実在する正式機能**（公式: code.claude.com/docs/en/workflows）。プロンプトに `ultracode` キーワードを入れる、または `/effort ultracode` で起動。Claudeがオーケストレーション台本（JavaScript）を書き、最大16並列・1回あたり最大1000エージェントで実行する。途中結果はコンテキスト窓に乗らないのでトークン窓を圧迫しにくい。注意点3つ：①同梱の `/deep-research` はWebSearchツール必須なので、jina運用のあなたは自作ワークフロー推奨 ②コストが重いのでまず1ディレクトリ・1質問で小さく試す（`/workflows` で消費を見ながら停止可）③保存先 `~/.claude/workflows/` はdotfiles管理に乗せられる。たとえば「複数トピックをjinaで横断検索→出典を相互チェック→1本に統合」のような重い調査を台本化できる。（元記事：9／公式裏取り済み）

### 📚 学習
7. **[P3] Managed Agents の設計思想を学ぶ（今は概念理解の段階）**：公式確認の結果、これは**API開発者向けのホスト型エージェント基盤**（`client.beta.sessions`、公式: anthropic.com/engineering/managed-agents）。普段のClaude Code CLIとは別レイヤーで、今すぐ導入する機能ではない。学ぶ価値があるのは「脳（Claude＋ハーネス）・手（サンドボックス）・**セッション（＝Claudeのコンテキスト窓ではなく、外部に永続化された追記専用ログ）**」を切り離す設計思想。session外部化の発想は、あなたのauto-memory運用やトークン節約（グループ1）の考え方と通じる。※記事が連想させる `/schedule`（Claude Codeのクラウド定期実行）とは**別物**。（元記事：2／公式裏取り済み）

## グループ3：Skills設計の6法則 — 既存12スキルの点検

項目1の引用記事が、100個のSkillsをリバースエンジニアリングして「動く/動かない」を分ける6法則を抽出しています。あなたは現在12個のスキルを運用中で、点検の対象がそろっています。

### 🛠 環境改善・向上
8. **[P2] 自作スキルを6法則の7項目チェックで点検する**：①descriptionにトリガーキーワード3つ以上 ②先頭250文字にユースケース ③命令形＋番号付きステップ ④出力形式の明示 ⑤実行前に既存ファイルを読む指示 ⑥やらないこと（Out of Scope）の明記 ⑦500行以内（入口SKILL.mdは50行目安）。1スキル15分。note-article-generate や x-bookmark 系など自作スキルから着手すると、自動呼び出しの精度が上がる。（元記事：1）
9. **[P3] 高品質スキルの70%が持つ「Out of Scope」記法を取り入れる**：「ここから先は別スキルへ」と境界を書くだけで誤発火が減る。skill-creatorスキルを持っているので、点検と改修を一緒に回せる。（元記事：1）

### 📚 学習
10. **[P2] 「descriptionは発注書、本文は命令形」の原則を血肉にする**：50文字未満のdescriptionは呼び出し頻度が3〜5倍下がるという実測。今後スキルを新規作成するときの土台になる考え方。（元記事：1）

### 🛠 環境改善・向上

### 📚 学習

### 📡 情報収集
